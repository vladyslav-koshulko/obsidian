# 🧩 Что такое **The Elm Architecture (TEA)**

**TEA** — это способ строить приложения как **чистый цикл обработки состояния**:

```
Model → View → Msg → Update → Model
```

---

### 🔁 Суть

1. **Model** — состояние (данные)
2. **View** — функция: Model → UI
3. **Msg** — события (клики, ответы API)
4. **Update** — функция: (Msg, Model) → Model

---

### Мини-пример

```
type Msg  
    = Increment  
    | Decrement  
  
update msg model =  
    case msg of  
        Increment -> model + 1  
        Decrement -> model - 1
```

👉 Никакой магии:

- нет мутаций
- всё через функции
- всё предсказуемо

---

## 🌳 Как это связано с Elm

Elm **построен вокруг TEA**.

👉 Это не просто паттерн — это **ядро языка**:

- компилятор заставляет тебя писать так
- архитектура встроена

---
## ⚙️ Где тут функциональное программирование

TEA = чистый FP в UI:

### 1. Чистые функции

- `update` — чистая
- `view` — чистая

### 2. Иммутабельность

- состояние не мутируется
- создаётся новое

### 3. Нет сайд-эффектов внутри логики

- эффекты вынесены отдельно

👉 это делает:

- код предсказуемым
- легко тестируемым

---

## ⚡ Где тут реактивное программирование

Реактивность = система **реагирует на события**

В TEA:

event → Msg → update → новый Model → новый UI

👉 это и есть реактивность

---

### Но важно

TEA ≠ классическое reactive programming

Сравнение:

|Подход|Как работает|
|---|---|
|TEA|явный цикл (Msg → Update)|
|Rx (например RxJS)|потоки (streams)|

---

## 🤝 Связь с React

React частично повторил идеи TEA:

### Похоже:

- state → UI
- декларативность

### Отличия:

- в React нет строгого цикла Msg → Update
- больше свободы → больше хаоса

---

## 🧠 Как это собрать в голове

- **Elm** — язык
- **TEA** — архитектура внутри него
- **FP** — философия (чистые функции)
- **Reactive** — поведение (реакция на события)

---

## 🚀 Практическое понимание (самое важное)

TEA решает 3 проблемы:

1. ❌ “где изменился state?”
2. ❌ “кто вызвал этот код?”
3. ❌ “почему UI сломался?”

👉 всё проходит через **одну точку — update**


# Из чего TEA состоит
Архитектура Elm (TEA) — это шаблон программирования, используемый в веб-приложениях для обработки интерактивных представлений и обновления состояний. Процесс очень прост: пользователь взаимодействует с пользовательским интерфейсом, что запускает событие, изменяющее состояние приложения, которое затем отражается в обновленном пользовательском интерфейсе.

	Рисунок 1 — Рабочий процесс ELM

Elm состоит из трех основных частей: Model, View и Update.

Модель **—** это часть приложения, которая служит в качестве снимка экрана, отображающего текущее состояние приложения. Она содержит данные для приложения, которые могут изменяться с помощью событий, например, переменная, определяющая количество нажатий кнопки.

Представление ( **View)** — это пользовательский интерфейс, генерируемый на основе текущего состояния модели. В предыдущем примере представлением могла бы быть HTML-страница, отображающая число и кнопку, где число — это значение переменной состояния. Это не обязательно должны быть отображаемые переменные; это может быть, например, страница, отображаемая только авторизованным пользователем, или список элементов, отображаемый на основе данных. По сути, представление — это то, что видит пользователь, и оно находится под влиянием текущей модели.

Обновление **(Update)** — является частью процесса и представляет собой функцию. Она отслеживает триггер от клиента (например, нажатие кнопки) и обрабатывает это сообщение о событии, возвращая обновленное состояние. При обработке сообщения функция обновления выполняет некоторую логику для вычисления/изменения состояния, например, добавляет единицу к количеству нажатий кнопки.

Эти основные компоненты станут основой при проектировании приложения Elm. Как показано на **рисунке 1** , мы видим, что потоки данных движутся в одном направлении: от представления к модели, а затем обратно к представлению.
